[FINDING AU-01] CRITICAL — No retry logic in collect_trends.py
  File   : scripts/collect_trends.py
  Lines  : 137–145 (safe_get), 251–263 (collect_google_trends)
  Detail : `safe_get()` makes a single attempt and returns None on any failure.
           The caller in `collect_google_trends()` records an empty list and sets
           `success = False`, but the pipeline continues without retrying.
           Google Trends RSS is known to return 429 (rate limit) or 5xx errors
           transiently. With 35 countries fetched sequentially, any one failure
           silently produces an empty trend array for that country.
  Impact : Countries can appear on the map with zero trends due to a transient
           network hiccup, with no human-visible alert.
  Fix    : Implement exponential backoff retry in safe_get() — e.g., 3 attempts
           with delays of 1s, 5s, 15s before giving up.

[FINDING AU-02] CRITICAL — No fallback / stale-data safety net
  File   : scripts/collect_trends.py, .github/workflows/collect-data.yml
  Lines  : collect_trends.py:672–673 (unconditional overwrite of OUTPUT_FILE)
  Detail : If the entire pipeline produces zero valid trends (all sources fail),
           the script still writes the output JSON, replacing the previous
           good data with an empty or near-empty dataset. The workflow has a
           "Verify output file" step that checks the file exists but does NOT
           check whether totalTrends > 0 before committing.
  Impact : A total API outage causes the live site to show an empty map and
           zero trend count, breaking user experience until the next hourly run.
  Fix    : Before overwriting trends.json, validate that totalTrends >= minimum
           threshold (e.g., 50). If below threshold, keep the existing file and
           exit with a warning rather than overwriting.

[FINDING AU-03] MEDIUM — Race condition between collect-data and generate-blog
  Files  : .github/workflows/collect-data.yml,
           .github/workflows/generate-blog.yml
  Detail : collect-data runs every hour (cron '0 * * * *'). generate-blog runs
           daily at midnight (cron '0 0 * * *'). At midnight, both workflows
           trigger simultaneously. generate-blog has a fallback to run
           collect_trends.py if trends.json is missing, but if collect-data is
           mid-run and has partially committed, generate-blog may read an
           incomplete JSON file or an older backup.
  Impact : Blog posts generated at midnight may be based on stale or
           partially-written trend data.
  Fix    : Add `needs: [collect]` dependency in generate-blog.yml (via
           workflow_call) or use a repository_dispatch event from collect-data
           to trigger blog generation only after a successful data commit.

[FINDING AU-04] MEDIUM — Failure alerting is GitHub-annotations-only
  Files  : .github/workflows/collect-data.yml (lines 75–79),
           .github/workflows/generate-blog.yml (lines 103–106)
  Detail : Both workflows use `echo "::error title=..."` on failure, which only
           creates GitHub annotations visible in the Actions UI. There is no
           external notification (email, Slack webhook, PagerDuty).
  Impact : If nobody checks the GitHub Actions dashboard, pipeline failures go
           undetected. With hourly runs, data could be stale for many hours
           before discovery.
  Fix    : Add a failure notification step using a Slack incoming webhook
           (stored as a GitHub secret) or GitHub's built-in email notifications
           for workflow failures.

[FINDING AU-05] MEDIUM — Wikipedia enrichment is bottlenecked by sequential I/O
  File   : scripts/collect_trends.py
  Lines  : 381–399 (fetch_wikipedia_description), 499–503 (enrich_trends loop)
  Detail : For each country's top-20 trends, if a description is missing, a
           Wikipedia API call is made synchronously with a 0.2s sleep.
           In the worst case: 35 countries × 20 trends × 0.2s = 140 seconds
           of pure sleep (plus actual network latency). This makes the
           collection pipeline potentially take 3–5 minutes on cold runs.
  Impact : The pipeline runs close to its 20-minute timeout in worst-case
           scenarios. Under heavy load or slow network, it may time out and
           produce no output.
  Fix    : Use concurrent.futures.ThreadPoolExecutor to fetch Wikipedia
           descriptions in parallel (10–20 threads). Reduce individual sleep
           to 0.05s. Add the _wiki_cache to avoid duplicate fetches.

[FINDING AU-06] LOW — generate_blog.py makes direct HTTP calls without a session
  File   : scripts/generate_blog.py
  Lines  : 221 (fetch_wiki_description), 232 (fetch_gdelt_headlines)
  Detail : Unlike collect_trends.py which uses a shared requests.Session(),
           generate_blog.py calls requests.get() directly each time, creating
           a new TCP connection per call. This is less efficient and bypasses
           any connection pooling.
  Impact : Slower execution and higher resource usage for blog generation runs
           that make many Wikipedia/GDELT calls.
  Fix    : Create a module-level SESSION = requests.Session() and use it for
           all outbound HTTP calls in generate_blog.py.
